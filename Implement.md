# Implement.md

## 1) 执行总则

- `Plan.md` 是唯一事实来源；如果实现与 Plan 冲突，先更新 Plan 再改代码。
- 每个里程碑结束必须立刻执行验证，验证失败不得进入下一里程碑。
- 严格控制 diff 范围：每个里程碑只改该里程碑定义的文件和职责。
- 持续更新文档：每次完成里程碑后同步更新 `Documentation.md` 状态和决策记录。

## 2) 标准执行流程（每个里程碑重复）

1. 对齐范围：
- 读取 `Plan.md` 当前里程碑边界、验收标准、验证命令。

2. 实施改动：
- 只改当前里程碑涉及文件，不顺手处理计划外问题。

3. 本地验证：
- 运行：`dotnet build UnityProject/Assembly-CSharp.csproj -nologo`
- 若里程碑包含运行态验收，进入 Unity Play Mode 执行手工验证步骤。

4. 记录结果：
- 在 `Documentation.md` 更新：
- 里程碑状态（`pending / in_progress / done / blocked`）
- 本次决策与原因
- 验证结果和发现的问题

5. 收口检查：
- 确认没有扩大任务边界。
- 确认日志开关、性能相关配置默认值符合 Prompt 约束。

## 3) 变更边界控制

- 允许：
- 当前里程碑定义内的代码、配置、文档修改。

- 不允许：
- 里程碑外功能重构。
- 未在 Plan 记录的“顺手优化”。
- 未验证就推进下一步。

## 4) 故障处理

- 如果编译失败：
- 立即回到最近一次可编译状态并修复，不继续叠加新改动。

- 如果行为回归（移动/攻击/寻路异常）：
- 先修功能正确性，再看性能。

- 如果性能回归：
- 先定位到具体函数和分配源，再做最小改动修复并重新验证。
