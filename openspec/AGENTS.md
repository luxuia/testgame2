# Agent Instructions (scope: `openspec/` and below)

## Scope and layout
- This AGENTS.md applies to: `openspec/` and below.
- Key directories:
  - `changes/` - active and archived change proposals/tasks
  - `changes/archive/` - completed historical changes
  - `specs/` - canonical capability specs
  - `config.yaml` - OpenSpec schema config (currently `spec-driven`)

## Commands
- If OpenSpec CLI is installed:
  - `openspec list --json`
  - `openspec status --change "<change-id>" --json`
  - `openspec instructions apply --change "<change-id>" --json`
  - `openspec validate --change "<change-id>" --strict` (if supported in your CLI build)
- Without CLI:
  - Edit artifacts directly under `changes/<change-id>/` and keep checklists/status consistent.

## Conventions
- Use kebab-case for change IDs (example: `optimize-pathfinding-gc`).
- For spec-driven flow, keep artifacts aligned:
  - `proposal.md` explains what/why.
  - `design.md` explains how.
  - `tasks.md` tracks implementation with `- [ ]` / `- [x]`.
- Archive only when implementation and task checklist are truly complete.
- Keep archived items immutable unless explicitly asked to reopen/correct history.

## Verification
- Confirm artifact completeness before implementation or archive:
  - status output shows required artifacts as done
  - `tasks.md` has no remaining unchecked items
- For manual review, check required files exist:
  - `.openspec.yaml` (if generated by CLI), `proposal.md`, `design.md`, `tasks.md`

## Common pitfalls
- Mixing active and archived changes in the same folder obscures progress; keep boundaries clean.
- Updating `specs/` without reflecting change context in `changes/` makes audits hard.
- Editing tasks without updating checkbox state causes false progress signals.

## Do not
- Do not place implementation code in `openspec/`.
- Do not modify `changes/archive/*` by default.
- Do not skip task/status updates after applying a change.
