## Context

当前项目中，命中、扣血、动画、控制禁用与对象删除分散在不同脚本里，尚未形成单一、可验证的生命周期流程。  
已有 `CombatEntityRuntime` 支持基础扣血，但死亡后的移动/控制禁用与删除回收尚未统一约束，容易出现“已死亡仍可被控制”或“对象删除后仍被寻路系统引用”的问题。  
本变更需要建立一个最小但完整的“单位战斗生命周期状态机”。

## Goals / Non-Goals

**Goals:**
- 建立统一命中结算入口，命中后必然进入扣血与状态更新。
- 当血量归零时立即进入死亡状态，并禁用移动和控制输入。
- 死亡后在可配置时机执行删除（Despawn），并完成引用清理。
- 提供事件钩子，便于动画、UI、掉落与后续系统订阅。

**Non-Goals:**
- 不实现复杂伤害类型体系（元素、穿透、抗性表）。
- 不引入完整掉落/经验/战斗结算系统，仅预留事件接口。
- 不重写寻路系统，只做死亡目标失效和清理接入。

## Decisions

1. 采用统一生命周期状态机  
单位状态最小集合：`Alive -> Dying -> Dead -> Despawned`。  
`Dying` 用于播放死亡动画与短暂收尾，`Dead` 期间移动和控制已禁用。  
理由：状态清晰且可扩展，不需要引入复杂行为树。

2. 命中与伤害结算集中到单入口  
所有攻击命中都走统一入口（例如 `ApplyHit/ApplyDamage`），不得绕开直接改血。  
理由：避免不同调用路径产生不一致死亡行为。

3. 死亡后的禁用策略  
进入 `Dying/Dead` 立即禁用：
- Player：输入控制与可移动逻辑  
- Agent：寻路目标推进与移动执行  
并屏蔽新的攻击输入。  
理由：确保“死亡即失能”语义可验证。

4. 删除策略采用“延时可配 + 默认即时/短延时”  
死亡后进入删除倒计时（可配置），到时执行对象删除和系统解绑。  
理由：兼顾动画演出与逻辑简洁；默认保持低成本。

5. 引用清理最小闭环  
删除前统一执行：
- 目标系统清理（当前目标置空）
- 分配/队列清理（从协调器或任务系统移除）
- 事件广播 `OnDespawn`  
理由：避免悬挂引用与空对象访问。

6. 事件模型保持轻量  
统一暴露：`OnHit`、`OnDamageApplied`、`OnDeath`、`OnDespawn`。  
理由：先保证可观察性，再逐步扩展掉落和UI系统。

## Risks / Trade-offs

- [风险] 不同单位脚本各自处理死亡可能产生分叉 → [缓解] 提供共享生命周期组件或公共接口，并统一调用。
- [风险] 删除时机过早导致动画被截断 → [缓解] 增加最短 `despawnDelay`，默认与死亡动画长度对齐。
- [风险] 删除时机过晚造成场景残留对象过多 → [缓解] 限制最大延时并提供批量清理兜底。
- [风险] 死亡后仍被寻路或目标系统引用 → [缓解] 在 `OnDeath` 和 `OnDespawn` 双阶段都执行失效与解绑。

## Migration Plan

1. 为 Player 与 CrowdAgent 接入统一死亡状态字段和禁用逻辑。  
2. 将现有伤害入口统一改为走生命周期接口。  
3. 接入删除倒计时与对象删除回调。  
4. 在协调器/目标系统增加死亡/删除目标清理。  
5. 回归验证：命中扣血、死亡禁用、删除后无悬挂引用。

## Open Questions

- 删除延时是否按单位类型区分（玩家、普通敌人、精英/BOSS）。  
- 玩家死亡是否走“删除对象”还是“保留对象并触发复活流程”（当前需求是删除，后续可能变化）。  
- `OnDeath` 到 `OnDespawn` 期间是否允许受击/治疗（建议默认不允许）。
