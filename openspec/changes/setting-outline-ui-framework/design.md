## Context

`doc/设定总纲.md` 已明确 UI 目标：MC 像素风、固定四组常驻核心模块、单侧边栏分页、统一弹窗模板，以及建造/经营/战斗一体化操作。当前工程已有 `ToaruUnity.UI` 作为页面栈与生命周期基础，但缺少项目级 UI 框架约束（分层、输入路由、状态桥接、页面边界），导致后续功能面板容易出现重复逻辑与交互割裂。

同时，项目当前的输入和核心系统（菌毯权限、战斗与 GOAP 行为、托管开关）已经存在，需要 UI 框架优先定义“如何接入”和“如何统一”，而不是先做大量页面细节。

## Goals / Non-Goals

**Goals:**
- 定义可落地的 UI 分层架构：常驻 HUD、侧边栏、弹窗、战斗浮层。
- 定义统一输入路由合同：左键/右键/WASD/Tab/E/Esc 在不同玩法阶段保持一致语义。
- 定义 UI 状态桥接合同：UI 只消费聚合态，不直接散落访问底层系统。
- 定义首版页面最小集合与优先级，支持分阶段交付。
- 保持与现有 `ToaruUnity.UI` 生命周期兼容，避免推翻已有基础。

**Non-Goals:**
- 不在本变更内完成全部 UI 美术资源制作。
- 不在本变更内一次性实现所有二级功能面板细节。
- 不重写底层战斗、寻路、区块与渲染系统。
- 不引入高复杂度的 UI 动画系统或新第三方 UI 框架。

## Decisions

1. 采用“四层 UI 容器 + 单入口管理”架构  
- 选择：建立 `UIRoot`，固定四个层级容器（`HUDLayer`、`SidePanelLayer`、`ModalLayer`、`CombatOverlayLayer`）。  
- 原因：与总纲的模块分组一致，避免页面层级混乱和遮挡冲突。  
- 备选：按功能模块各自创建 Canvas。被拒绝，因为会增加渲染与交互管理复杂度。

2. 输入采用“命令路由”而非“页面直连输入”  
- 选择：先把原始输入映射到统一 UI 命令，再由当前上下文（是否选中单位、是否打开面板、是否处于战斗）分发。  
- 原因：保证“单套按键跨玩法”可持续维护，避免页面互相抢输入。  
- 备选：每个页面单独监听输入。被拒绝，因为容易造成语义冲突。

3. 页面状态采用“聚合态 + 单向刷新”  
- 选择：以 `ActionCenter/ActionState` 为基础，建立项目级 UI 聚合状态层（如菌主状态、快捷栏状态、托管状态、警报状态）。  
- 原因：减少页面直接依赖领域系统，降低联动改动成本。  
- 备选：页面直接读取 `FungalCarpetSystem`/`CombatFramework`。被拒绝，因为耦合高且难测试。

4. 首版页面采用“骨架优先、内容渐进”策略  
- 选择：先交付可切换可显示的框架页与占位数据，再分批接入真实数据和交互细节。  
- 原因：首版目标是建立稳定框架而非一次性堆满内容，符合小团队低成本节奏。  
- 备选：直接完成所有页面内容。被拒绝，因为风险高、回归成本大。

5. 视觉样式采用“单套像素模板资产”策略  
- 选择：边框、按钮、进度条、列表项采用统一像素模板，页面按模板复用。  
- 原因：符合总纲“1 套 16x16 素材模板复用”的降本目标。  
- 备选：每个页面独立皮肤。被拒绝，因为美术成本不可控。

6. 战斗 UI 采用“事件驱动显隐”  
- 选择：战斗开始显示 `CombatOverlay`，战斗结束自动隐藏；共鸣条件与 BOSS 机制采用统一事件触发。  
- 原因：满足总纲“战斗时出现、结束即收起”的要求，减少常驻噪音。  
- 备选：战斗信息常驻显示。被拒绝，因为会增加信息过载。

## Risks / Trade-offs

- [风险] 输入语义冲突导致误操作（如右键在不同状态行为不一致） -> 缓解：建立命令优先级表与冲突回归用例。  
- [风险] 状态聚合层字段膨胀 -> 缓解：按模块拆分聚合态（HUD/SidePanel/Combat），只暴露只读快照。  
- [风险] 页面骨架先行导致“看得到但不可用”的体验落差 -> 缓解：为占位内容明确标注并优先补齐高频路径。  
- [风险] 多层 Canvas 叠加引发性能与点击穿透问题 -> 缓解：限制层级数量、统一 Raycast 规则、建立基础性能基线。

## Migration Plan

1. 创建 `UIRoot` 与四层容器，接入 `SinglePlayer` 场景入口。  
2. 建立 UI 命令路由与状态聚合接口，替换散落输入监听。  
3. 落地首版页面骨架：HUD、侧边栏、弹窗模板、战斗浮层。  
4. 对接最小真实数据：菌主状态、快捷栏、托管状态、战斗告警。  
5. 完成回归验证后，再分批填充二级面板详细内容。  
6. 回滚策略：若出现重大问题，保留旧 UI 入口开关并按层级逐个回退。

- 侧边栏分页在新手模式下的解锁节奏是否由数据配置还是剧情脚本驱动。  
- 战斗浮层中的 BOSS 文本提示是否需要分级颜色规范（预警/危险/致命）。  
- 右下角固定按钮在手柄布局下的交互提示文案是否需要单独模板。
