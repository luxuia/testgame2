## Context

当前代码路径中，`TargetSelector` 与 `PlayerController` 以单目标方块为核心，`ShaderUtility` 也只暴露单个 `_TargetBlockPosition`。  
同时，`CrowdFlowFieldCoordinator` 已具备多 agent 目标分配、共享目标场流、阵型分散与局部避障能力，可复用为“挖掘执行层”的移动基础。  
本变更需要把“单目标点击挖掘”升级为“3D 框选任务驱动 + 槽位容量约束 + 协作挖掘”。

## Goals / Non-Goals

**Goals:**
- 支持点击单块和拖拽 3D 体积框选两种挖掘入口。
- 选中方块保持清晰可视，且与现有高亮体验兼容。
- agent 按方块周围可站位数量分配，避免无效拥挤。
- 同一方块支持多 agent 线性叠加挖掘伤害。
- 在大选区下控制计算峰值和 GC 分配。

**Non-Goals:**
- 不在本次变更中重写 FlowField/A* 底层算法。
- 不引入复杂施工优先级系统（例如资源价值、路径代价全局最优调度）。
- 不修改已有材质系统的底层渲染管线结构。

## Decisions

1. 输入与选区模型  
采用“按下记录起点、抬起确定终点”的块坐标体积框选模型。  
点击与拖拽通过屏幕位移阈值区分；低于阈值按单块处理。  
理由：与现有 raycast 方块选中机制兼容，实现成本低且稳定。

2. 选中可视化策略  
保留现有 `_TargetBlockPosition + _DigProgress` 作为“焦点块”表现；其余选中块使用对象池 marker 批量高亮。  
理由：避免一次性改 shader 为大数组输入，降低渲染/CPU复杂度与风险。

3. 任务数据模型  
新增 `MiningJobManager` 聚合四类状态：
- SelectedBlocks: 选中方块集合
- BlockJobs: 每块挖掘状态（剩余硬度、参与者）
- Slots: 每块可站位槽位集合
- Assignments: agent -> slot -> block 映射  
理由：将输入、分配、执行解耦，便于增量更新和后续扩展。

4. 可站位判定口径  
槽位可用性统一复用 pathfinding walkable 判定（与 `AStarPathfinding.IsWalkableNode` 一致）。  
理由：减少“可分配但不可达”的规则不一致问题。

5. 分配算法  
采用两轮分配：
- 第一轮：尽量让更多 block 至少有 1 个 agent。
- 第二轮：再填充同块剩余可站位槽位。  
同时采用稳定分配：保持既有 assignment，只有在方块完成/槽位失效/agent失效时局部重排。  
理由：满足“10块尽量10人各挖一块”的期望，并减少抖动和重算。

6. 移动目标策略  
中远距离沿用现有 coordinator 的路径采样能力；近距离进入槽位接管。  
批量场景下优先按 block 级别聚合目标，避免每个 agent 独立 target 引发 `PrepareFields` 爆炸。  
理由：兼顾路径质量和性能目标。

7. 协作伤害模型  
单块挖掘总伤害为所有有效参与 agent 的线性和：`sum(agentDps) * deltaTime`。  
理由：规则直观，玩家反馈明确，易于调参与验证。

## Risks / Trade-offs

- [风险] 大体积框选导致任务创建与槽位计算峰值过高 -> [缓解] 增加 `MaxSelectedBlocks` 与分帧构建。
- [风险] 选中块过多时高亮对象数量大 -> [缓解] marker 对象池 + 距离裁剪 + 上限保护。
- [风险] 频繁重分配导致 agent 抖动 -> [缓解] 稳定分配、局部重排、重分配冷却。
- [风险] 多人线性叠加可能造成单块瞬时完成过快 -> [缓解] 通过 agent DPS 系数和上限做平衡。

## Migration Plan

1. 先接入新任务层，但保留单目标流程兜底开关。  
2. 在测试场景验证：
- 点击单块回归行为
- 3D 框选正确率
- 1块2槽仅2人参与
- 10块多agent分散施工
3. 通过后将默认入口切到任务层，并保留调试开关。
