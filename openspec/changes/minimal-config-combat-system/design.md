## Context

《设定总纲》要求战斗系统必须与地形改造强绑定，并明确区分菌毯主场与无菌毯客场的规则。  
当前项目已具备输入、寻路、基础交互能力，但尚缺战斗领域的统一约束，尤其是“权限判定 + 托管保底 + BOSS 地形互动 + 最小配置项”这一组核心机制。  
本设计目标是在不重写底层系统的前提下，先建立可实施、可验证、可扩展的战斗 MVP 基线。

## Goals / Non-Goals

**Goals:**
- 建立统一的战斗上下文模型，支持主场/客场差异化规则。
- 将地形编辑权限作为战斗规则的一部分，而非独立系统。
- 将配置面约束到 9 个全局参数，避免参数爆炸。
- 提供托管保底与失败自适应机制，保证新手可通关路径。
- 提供低成本但可感知的 BOSS 地形互动机制。

**Non-Goals:**
- 不在本次变更中重写寻路、网格、区块存储等底层架构。
- 不实现完整七罪技能树与全部噗叽类型平衡，只定义可承载其扩展的最小骨架。
- 不引入高成本演出系统（复杂镜头、重特效、专用编辑器）。

## Decisions

1. 战斗上下文采用“领地权限优先”模型  
将战斗发生区域标记为 `Home`（菌毯覆盖）或 `Away`（非菌毯覆盖）。所有动作先经过权限网关判定，再进入执行。  
理由：直接满足“菌毯即权柄”设定，且实现上只需复用方块属性与区域查询。  
备选方案：按玩法模式切换权限（建造模式/战斗模式）被拒绝，因为会增加模式切换复杂度并破坏一套操作通吃原则。

2. 统一动作管线  
普攻、技能、闪避、破坏方块、放置方块均走统一 Action 执行链（输入 -> 校验 -> 执行 -> 结果事件）。  
理由：减少重复逻辑，便于托管与手动共用一套能力定义。  
备选方案：地形编辑独立于战斗执行链被拒绝，因为会造成规则分叉和维护成本上升。

3. 主客场规则最小化  
主场提供固定攻防增益并开放自由编辑；客场默认关闭自由编辑，仅可使用随身菌毯道具解锁临时小范围编辑。  
理由：既保留策略差异，又不引入复杂权限矩阵。  
备选方案：客场完全禁止编辑被拒绝，因为不符合总纲中的“客场可临时搭掩体/挖地道”。

4. 托管保底采用阈值触发而非行为树重构  
当核心生命低于阈值时触发固定策略：副官回防 -> 噗叽修复优先 -> 防御建造优先。  
理由：规则可解释、实现成本低、可通过少量参数调优。  
备选方案：全量行为树/GOAP 托管被拒绝，因为超出小团队成本预算。

5. BOSS 地形互动采用“周期破坏 + 低血狂暴”两段式  
阶段一：固定间隔破坏掩体。阶段二：低血量后提升破坏频率与攻速。  
理由：有明显战斗节奏变化，且几乎不需要额外系统依赖。  
备选方案：每个 BOSS 定制复杂地形机制被拒绝，因为内容产能风险高。

6. 配置面严格封顶为 9 项  
MVP 仅暴露以下全局参数：  
- `homeDamageBonusPct`  
- `homeDefenseBonusPct`  
- `awayEditCharges`  
- `awayEditRadius`  
- `dodgeInvulnSec`  
- `autoTakeoverCoreHpPct`  
- `autoRepairIntervalSec`  
- `bossTerrainBreakIntervalSec`  
- `adaptiveFailCountThreshold`  
其余规则采用固定公式与约定默认值。  
理由：降低联调复杂度与策划负担，保证“最小化配置项”目标可验收。  
备选方案：按单位/技能/场景开放大量可调参数被拒绝，因为会显著增加维护和测试组合爆炸。

7. 技能系统采用统一数据结构与固定执行阶段  
所有单位技能统一为：`Trigger + Targeting + Phases + Effects + Cooldown/Cost`。  
执行阶段固定为：`Windup -> Execute -> Recovery`。  
效果类型在 MVP 限定为：`Damage`、`Heal`、`BuffDebuff`、`BreakBlock`、`PlaceBlock`、`Summon`。  
理由：降低实现分叉，确保手动操作、托管执行、敌方施法共用同一执行管线。  
备选方案：按单位类型分裂多套技能执行器被拒绝，因为会显著增加维护与调试成本。

8. AI 采用“三层调度 + 轻量决策”  
- `BattleDirector`：全局态势与优先级切换（攻防、回防、修复）。  
- `RoleBrain`：单位级状态决策，基于有限状态机（FSM）和轻量评分。  
- `ActionExecutor`：动作与技能实际执行，复用统一 Action 管线。  
标准状态集合：`Idle`、`Move`、`Attack`、`TerrainOp`、`Retreat`、`Repair`。  
理由：比行为树/GOAP 更低成本，且足以覆盖主客场战斗与托管保底。  
备选方案：全量行为树被拒绝，因为对当前团队规模和迭代节奏不经济。

9. 配置采用“全局键封顶 + 模板扩展”两层策略  
- 阶段 A（MVP）：仅开放 9 个全局参数。  
- 阶段 B（内容扩展）：不新增全局参数，新增模板配置：`SkillPreset`、`AIPreset`、`EncounterPreset`。  
- 阶段 C（必要扩展）：若必须新增全局键，先更新 OpenSpec 再扩配置。  
理由：先保稳定与可控，再通过模板提升内容生产效率，避免参数爆炸。

## Risks / Trade-offs

- [风险] 固定 9 参数可能限制早期数值调优灵活度 -> [缓解] 保留内部常量分组，后续仅在确认必要时增量外露参数。
- [风险] 客场临时编辑权限不足可能导致体验受挫 -> [缓解] 通过 `awayEditCharges` 与 `awayEditRadius` 快速调节容错。
- [风险] 托管阈值触发后行为可能显得“机械” -> [缓解] 优先保证保底通关，后续再增加行为细化层。
- [风险] BOSS 两段式机制重复感较强 -> [缓解] 在不新增系统复杂度前提下，通过不同破坏目标偏好实现差异化。
- [风险] FSM + 评分在复杂战场下可能出现短时抖动 -> [缓解] 对状态切换加最短驻留时间与冷却窗口。
- [风险] 技能模板复用初期可能限制个别单位特色 -> [缓解] 在不改执行骨架前提下，通过 `SkillPreset` 组合差异化。

## Migration Plan

1. 先在 OpenSpec 中固化规格和任务，冻结 MVP 边界。  
2. 实现阶段按“基础战斗 -> 权限判定 -> 托管保底 -> BOSS 机制 -> 验证”顺序渐进接入。  
3. 在 `SinglePlayer` 场景进行回归，重点验证主客场切换、地形动作权限、托管触发和 BOSS 机制。  
4. 首次实现同时落地 `SkillPreset` 与 `AIPreset` 的最小样例，验证模板配置可驱动至少玩家/副官/BOSS 三类角色。  
5. 若关键规则不可用，回滚到“仅基础战斗 + 无托管 + 无BOSS地形互动”保底版本，同时保留规格不变用于后续补齐。

## Open Questions

- 客场临时菌毯的持续时间是否固定，还是与技能/道具等级挂钩。  
- 托管触发后是否需要短暂冷却，避免阈值附近频繁抖动开关。  
- 主场攻防增益是否应随菌毯完整度变化，还是保持固定百分比。
