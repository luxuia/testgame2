## Context

上一变更 `setting-outline-ui-framework` 解决的是 UI 架构与规则定义，本次关注“对应 Prefab 制作”落地。当前项目虽有 `ToaruUnity.UI` 运行时基础，但场景内缺少规范化的 UI 预制体资产集合，导致页面难以复用、层级容易混乱、后续接入成本高。

设定总纲对 UI 结构已给出明确目标：常驻核心 HUD、单侧边栏、统一二级弹窗、战斗浮层。为了让开发节奏可控，必须把这些目标转化为可交付、可验收的 Prefab 资产与装配规范。

## Goals / Non-Goals

**Goals:**
- 交付一套可复用 UI 预制体骨架，覆盖 `UIRoot/HUD/SidePanel/Modal/CombatOverlay`。
- 建立统一 Prefab 层级、命名、挂载与引用规范，避免场景重复搭建。
- 确保 Prefab 可在 `SinglePlayer` 场景一键挂载并正常打开关闭。
- 为后续功能数据接入预留稳定绑定节点，减少返工。

**Non-Goals:**
- 不在本次完成全部页面真实业务数据与交互细节。
- 不在本次完成最终美术风格打磨与全量动画效果。
- 不更换或重构 `ToaruUnity.UI` 底层实现。
- 不扩展与 UI Prefab 无关的战斗/寻路/世界逻辑。

## Decisions

1. Prefab 采用“主根 + 子模块”组织  
- 选择：创建 `UIRoot.prefab` 作为唯一入口，子节点固定四层容器；核心页面作为独立子 Prefab。  
- 原因：便于场景挂载、版本管理与差异化替换。  
- 备选：所有页面都直接挂场景。被拒绝，因为可复用性差且容易漂移。

2. 页面资源采用“模板化壳 + 内容区”策略  
- 选择：`ModalTemplate` 提供统一边框/关闭/标题槽，具体功能页只填内容区。  
- 原因：匹配设定总纲的低成本复用要求，减少重复制作。  
- 备选：每种二级面板单独完整 Prefab。被拒绝，因为重复量过高。

3. Prefab 绑定采用“必要脚本最小化”  
- 选择：预制体仅绑定必需的视图脚本、路由入口、占位数据适配器，不放复杂业务逻辑。  
- 原因：让 Prefab 作为表现与装配层，降低逻辑耦合。  
- 备选：把业务逻辑直接写入页面脚本。被拒绝，因为后续维护风险高。

4. 场景集成采用“显式入口对象”  
- 选择：在 `SinglePlayer` 放置显式 `UIBootstrap`（或等效入口）引用 `UIRoot`。  
- 原因：可追踪、可替换、可做运行时校验。  
- 备选：依赖隐式自动查找。被拒绝，因为初始化顺序不稳定。

5. 验收采用“可实例化 + 可切换 + 无缺引用”三段式  
- 选择：Prefab 验收最小标准包含：实例化成功、核心页面可开关、控制台无 Missing Reference。  
- 原因：这是最直接且低成本的交付质量门槛。  
- 备选：仅靠视觉截图验收。被拒绝，因为无法保证运行可用性。

## Risks / Trade-offs

- [风险] Prefab 过早固化导致后续需求变更成本上升 -> 缓解：先定义稳定插槽与扩展点，细节内容放后续迭代。  
- [风险] 子 Prefab 过多引发引用链复杂 -> 缓解：限定层级深度与命名规范，避免跨层互引。  
- [风险] 场景中出现重复 UIRoot 实例 -> 缓解：入口启动时做单例检测并阻断重复创建。  
- [风险] 占位数据与真实数据切换时出现字段不兼容 -> 缓解：先定义统一 ViewModel 契约，字段变更走版本迁移。

## Migration Plan

1. 建立 `Assets/Prefabs/UI/` 目录规范与命名规则。  
2. 制作 `UIRoot.prefab` 与四层容器，接入 `ToaruUnity.UI` 管理入口。  
3. 制作首批核心页面 Prefab：`HUDRoot`、`SidePanel`、`ModalTemplate`、`CombatOverlay`。  
4. 在 `SinglePlayer` 场景挂载 `UIBootstrap` 并引用 `UIRoot`。  
5. 执行基础验收：页面开关、层级遮挡、控制台缺引用检查。  
6. 通过后再扩展二级内容页并逐步替换占位节点。
